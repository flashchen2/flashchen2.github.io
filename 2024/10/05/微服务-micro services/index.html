微服务

一、RPC

1.什么是RPC

  • RPC（Remote Procedure Call Protocol）远程过程调用协议。一个通俗的描述是：客户端在不知道调用细节的情况下，调用存在于远程计算机上的某个对象，就像调用本地应用程序中的对象一样

  • rpc的调用过程

2.为什么要用RPC

当我们的业务越来越多、应用也越来越多时，自然的，我们会发现有些功能已经不能简单划分开来或者划分不出来。此时，可以将公共业务逻辑抽离出来，将之组成独立的服务Service应用 。而原有的、新增的应用都可以与那些独立的Service应用 交互，以此来完成完整的业务功能

  1. 服务化：微服务化，跨平台的服务之间远程调用；
  2. 分布式系统架构：分布式服务跨机器进行远程调用；
  3. 服务可重用：开发一个公共能力服务，供多个服务远程调用。
  4. 系统间交互调用：两台服务器A、B，服务器A上的应用a需要调用服务器B上的应用b提供的方法，而应用a和应用b不在一个内存空间，不能直接调用，此时，需要通过网络传输来表达需要调用的语义及传输调用的数据。

3.常见的RPC框架

  1. GRPC是业界常用的，Google 开发的具体 RPC 框架

  2. Ks在GRPC的基础上，开发了KRPC

4.RPC调用流程

RPC的目标就是要2~8这些步骤都封装起来，让用户对这些细节透明。

  1. 服务消费方（client）调用以本地调用方式调用服务；
  2. client stub接收到调用后负责将方法、参数等组装成能够进行网络传输的消息体；
  3. client stub找到服务地址，并将消息发送到服务端；
  4. server stub收到消息后进行解码；
  5. server stub根据解码结果调用本地的服务；
  6. 本地服务执行并将结果返回给server stub；
  7. server stub将返回结果打包成消息并发送至消费方；
  8. client stub接收到消息，并进行解码；
  9. 服务消费方得到最终结果。

二、服务加固

1.链路梳理

1. 链路说明： a -> b -> c 1. 若只关注 a -> b。a为上游主调，b为下游被调 2. 若关注 a -> b -> c。也可以看做a为主调，b、c都为a的被调。但一般不会这样做，而是逐级的去看b是a的被调，c是b的被调 3. 被调链路可以看做是，主调链路的一个依赖。其中存在强依赖、弱依赖

1.超时

  1. 加固策略：
    1. 超时是站在主调角度去配置
    2. 超时时间：一般根据P99_rt的1.2倍进行配置

2.熔断

  1. 功能介绍

    1. 熔断是一种保护机制，用于在服务调用过程中，当某个服务或资源出现故障时，暂时停止对该服务或资源的调用，降低下游请求量，以防止故障扩散，保护系统的整体稳定性。
    2. 熔断机制通常用于处理弱依赖服务，当这些服务不可用时，可以通过熔断机制快速失败，避免影响主调服务的正常运行。

  2. 架构设计

    1. 单位时间内，窗口内统计的失败率，超过熔断触发阈值后。进入熔断Open状态，暂停请求下游服务
    2. 进入熔断状态，等待x时间后。尝试发起少量请求，进入Half Open状态，放出一部分请求去探测
    3. 如果探测失败率还是>熔断阈值，则再进入Open状态，等待下一次探测，否则进入Closed状态

  3. 注意事项

    1. 只有弱依赖可以设置熔断
    2. 熔断后，项目会走降级策略，走兜底逻辑
    3. 熔断是站在主调角度去配置
    4. 主调熔断是单机维度

3.限流

  1. 功能介绍

    1. 限流是指对请求的数量或流量进行控制，目的是保护系统资源，确保服务的稳定性与可用性，防止因过载导致的服务崩溃或性能下降

  2. 加固策略

    1. 限流是站在被调角度去配置
    2. 限流值：根据监控数据，可取近15天，峰值QPS的1.2倍进行配置

  3. 单机限流和集群限流

    1. 限流值计算方式

      1. 单机限流：

        • 被调限流配额在单机生效，被调所属环境的整体限流配额=单机限流配额*实例数

        • 扩缩容时，单机限流阈值不变化

      2. 集群限流：

        • 被调所属环境所有实例的整体限流值，限流逻辑按照“配额➗实例数”转换成单机限流配额

        • 扩缩容时，单机限流阈值会发生变化

    2. 两种限流方式特点

      1. 单机限流：

        • 保护自身服务稳定运行

        • 在高并发情况下，集群扩容后，集群整体流量可能激增，对下游服务造成压力，可能会影响整个系统的可用性

      2. 集群限流

        • 保障下游链路在高并发情况下能够稳定运行，避免因某个节点过载而影响整个系统的可用性
        • 可能存在流量分布不均的情况，导致部分单机服务承受压力过大而崩溃
